Troubleshooting

This page centralizes common errors from the project READMEs and gives immediate checks and fixes for staff supporting course teams.

Authentication and CORS Issues

If login behavior seems inconsistent, start here first. Cookie handling and cross-origin configuration are the most common causes of “works locally, fails in cloud” behavior.

Symptom Likely Cause Fix
403 /auth/test on first load No auth cookie yet Expected before login; complete login first.
Login succeeds but user appears logged out Credentials not sent with requests Use withCredentials: true (Axios) and credentials: include (fetch).
CORS or cookie errors Frontend URL mismatch or credentials disabled on server Set backend CORS origin to exact frontend URL and enable credentials on both sides.
404 /login from backend Missing auth route prefix Call /auth/login and /auth/register.

Database and SSL Issues

Database errors often look like application failures. Validate SSL certificate path, credentials, and network access before changing route or UI code.

Symptom Likely Cause Fix
ER_NO_DEFAULT_FOR_FIELD 'email' Request missing required email or schema mismatch Send email in register payload and verify users table schema.
MySQL TLS connection failure Missing or wrong CA path Place Azure certificate at config/DigiCertGlobalRootG2.crt.pem.
connect ETIMEDOUT to MySQL on Azure Network or firewall path issue Confirm server access settings and escalate to instructor or WVU cloud admin support.
Cannot connect from local tools Client IP not allowed Add or update Azure MySQL firewall rule for your public IP.

GitHub Actions and Deployment Issues

Pipeline failures are usually path, script, or secret mismatches. Compare workflow file values directly to your repository folder names and secret names.

Symptom Likely Cause Fix
Workflow cannot find backend directory Wrong path like ./backend Use sprint4-cloud-deployment/backend in working-directory, upload path, and package path.
Backend workflow fails at test step Default test script exits with status 1 Disable placeholder test command or replace with real passing tests.
Frontend build points at wrong API URL VITE_BACKEND_API_URL not set in workflow env Add secret and map it to deploy job environment.
SWA YAML validation warning in close PR job Missing app_location in close job config Add same app_location value as main deploy job.

App Service Runtime Issues

  • If backend does not start on Azure, confirm app listens on process.env.PORT.
  • If health endpoint fails, confirm App Service application settings were saved and app was restarted.
  • If frontend calls fail in production, verify FRONTEND_URL in backend settings matches SWA URL exactly.
  • Use App Service Log Stream while restarting app to capture startup errors immediately.

Fast Triage Workflow for Staff

  1. Check GitHub Actions status for the latest backend and frontend deployments.
  2. Validate backend health endpoint in browser.
  3. Open browser dev tools and confirm frontend requests target correct backend URL.
  4. Review App Service log stream for startup, CORS, and DB errors.
  5. If resource or network-level issue is suspected, escalate through WVU support process.
Escalation note: When missing resource groups or restricted cloud access blocks progress, route the request to Statler ITS.
