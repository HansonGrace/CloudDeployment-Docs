End-to-End Deployment Steps

Use this as the single sequence for professors and staff supporting class teams. It combines local setup, integration, CI/CD, and production validation into one ordered process.

WVU note: If cloud resources are missing, submit a request to Statler ITS for resource group and access support before continuing.

Step-by-Step Runbook

Follow these steps in order. If a step fails, resolve it before moving ahead so later cloud checks are based on a known-good state.

  1. Confirm team has required software: Node.js, Git, VS Code, and Azure/GitHub access.
  2. Confirm WVU resource group and Azure resources exist (request via Statler ITS if needed).
  3. Clone repository, open project, and verify baseline structure.
  4. Set up backend locally: install packages, configure .env, add SSL cert, and run backend.
  5. Set up frontend locally: install packages, configure backend URL env variable, and run Vite.
  6. Create database/table if not already prepared and verify authentication routes work.
  7. Validate Sprint 3 integration path: date selection, STAC fetch, and map visualization.
  8. Add required GitHub secrets for backend publish profile and frontend backend URL.
  9. Inspect backend workflow paths and deploy to App Service on push to main.
  10. Set App Service application settings and verify /health endpoint returns success.
  11. Inspect SWA workflow settings and deploy frontend through GitHub Actions.
  12. Perform end-to-end production tests: login, map fetch, and backend request validation from browser network tools.

Commands Used Most Often

These are the core commands used during setup and redeployment. If you cloned https://github.com/tdevine1/xPostForecast, these paths should work as written. If you are using your own repository, update folder paths to match your project structure. 

# Backend local run
cd sprint2-login-backend/backend
npm install
node server.js

# Frontend local run
cd sprint2-login-backend/frontend
npm install
npm run dev

# Trigger cloud deployment
git add .
git commit -m "Deploy updates"
git push origin main

Critical Deployment Checks

Use this section as a quick side-by-side review when a deployment succeeds in one layer but fails in the other. Most production issues come from one missed setting in these lists.

Backend (App Service)

  • Workflow path uses sprint4-cloud-deployment/backend
  • APP_SERVICE_PUBLISH_PROFILE secret is present
  • PORT is read from environment at runtime
  • App settings contain DB, JWT, and frontend URL values
  • /health returns healthy JSON response

Frontend (Static Web App)

  • app_location points to Sprint 4 frontend folder
  • output_location set to dist
  • VITE_BACKEND_API_URL secret is provided
  • Build succeeds in GitHub Actions
  • Browser requests hit App Service backend endpoints

Final Acceptance Checklist

Consider the deployment complete only after all items below are true in the live environment, not just in local development.

  • Frontend site loads at SWA URL without build-time runtime errors.
  • Users can register and log in using production backend.
  • Session verification route works using cookie-based auth.
  • Temperature data requests return points and map renders markers.
  • No CORS or mixed-origin issues in browser console.
  • App Service log stream does not show startup crashes or DB connection failures.
