Getting Started

This page is the starting point for WVU professors and staff. It combines Sprint 1 through Sprint 3 setup details into one clear foundation before cloud deployment.

WVU resource group process: At WVU, project teams request Azure resource group setup through Statler ITS. Do this first so deployment resources are available.
Next: GitHub Setup

Prerequisites

Before you begin, make sure each person supporting the course has both local developer tooling and the correct cloud/repository access. Most deployment blockers come from missing permissions.

  • Node.js 20.19+ and npm
  • Git
  • VS Code
  • GitHub repository access for your course project
  • Azure access details from instructor or WVU support staff

Recommended VS Code tooling from the READMEs: MySQL extension by Microsoft or SQLTools with MySQL driver.

Choose Your Starting Path

You can either clone the course reference repository directly or follow the same documentation for your own project repository. Both paths work, but cloning the reference project is the fastest way for new staff to practice the full deployment sequence.

Option A: Clone xPostForecast (recommended)

Use this if you want the exact reference structure used in class documentation.

git clone https://github.com/tdevine1/xPostForecast.git
cd xPostForecast
git checkout main
git pull origin main

If you forked first, keep your fork updated:

git remote add upstream https://github.com/tdevine1/xPostForecast.git
git fetch upstream
git checkout main
git merge upstream/main
git push origin main

Option B: Use your own project

Use this if your class team already has a separate repo. Keep the same architecture pattern: React frontend, Node/Express backend, MySQL database, and Azure deployment targets.

You can still follow every page in this guide; just adapt folder paths and environment variable names to your repository layout.

Reference Architecture

The project is intentionally split so each layer can be tested independently. Frontend issues, backend issues, and database issues can be isolated quickly when this structure is preserved.

Frontend

  • React + Vite app
  • Login/register pages and map page
  • Axios client in src/api.js
  • Leaflet map for WV temperature points

Backend and Data

  • Node.js + Express backend
  • JWT auth in HTTP-only cookies
  • Azure Database for MySQL with SSL cert
  • STAC data pipeline from Planetary Computer

Initial Local Setup

The sequence below assumes you are validating that the app works locally before cloud deployment. Completing this successfully prevents most CI/CD failures later.

  1. Clone the project repository and open it in VS Code.
  2. Install frontend dependencies and run Vite locally.
  3. Install backend dependencies and configure backend .env.
  4. Download Azure MySQL SSL certificate and place it in backend config.
  5. Create database and users table, then run backend.
  6. Configure frontend .env with backend URL and test login flow.
# Example using the reference repository
git clone https://github.com/tdevine1/xPostForecast.git
cd xPostForecast

# frontend
cd sprint2-login-backend/frontend
npm install
npm run dev

# backend (in another terminal)
cd sprint2-login-backend/backend
npm install
node server.js

Backend Environment Variables (Local)

Use the exact variable names from your project branch .env.example. Small naming differences (for example DB_PASS vs DB_PASSWORD) are a common source of configuration bugs.

Variable Purpose
DB_HOST Azure MySQL host, usually <server>.mysql.database.azure.com
DB_USER, DB_PASSWORD or DB_PASS Database credentials
DB_NAME or DB_DATABASE Database name for your class team
DB_PORT Typically 3306
FRONTEND_URL Local frontend origin, usually http://localhost:5173
JWT_SECRET JWT signing secret

Database Bootstrap SQL

From the Sprint 2 backend guide, run this against Azure MySQL to prepare authentication:

CREATE DATABASE IF NOT EXISTS authdb;
USE authdb;

CREATE TABLE IF NOT EXISTS users (
  id INT AUTO_INCREMENT PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE,
  username VARCHAR(255) NOT NULL UNIQUE,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Minimum Validation Before Cloud Work

Treat this as a gate: if any item below fails locally, pause and fix it before moving to GitHub Actions or Azure deployment. This saves significant debugging time in production.

  • Frontend loads on http://localhost:5173.
  • Backend starts with no database connection errors.
  • /auth/register, /auth/login, and /auth/test are working.
  • Map page and date selector render after login.
