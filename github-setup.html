Repository Baseline

Start by confirming how your team will track changes. If you are using the reference project, keep your local copy synchronized with the upstream repository so deployment docs and code stay aligned.

  1. Confirm instructors, staff, and project teams all have the correct repo permissions.
  2. Use main as deployment branch unless course instructions say otherwise.
  3. Ensure a valid .gitignore excludes .env, node_modules, and dist.
  4. Use pull requests for major deployment changes when possible.
# Clone the reference project
git clone https://github.com/tdevine1/xPostForecast.git
cd xPostForecast
git checkout main
git pull origin main

# If using a fork, sync with upstream before deployment work
git remote add upstream https://github.com/tdevine1/xPostForecast.git
git fetch upstream
git merge upstream/main
git push origin main

# Standard update cycle
git status
git add .
git commit -m "Update deployment configuration"
git push origin main

Required GitHub Secrets

Secrets are required because Azure deployments need credentials and environment values that should never be committed to source control.

Secret Name Used By Value
APP_SERVICE_PUBLISH_PROFILE Backend App Service workflow Full XML contents of App Service publish profile
VITE_BACKEND_API_URL Frontend SWA workflow build environment https://<backend-app>.azurewebsites.net (no trailing slash)
AZURE_STATIC_WEB_APPS_API_TOKEN_* Auto-generated SWA deployment token Created by Azure when SWA is linked to GitHub

Backend Workflow Checks

For backend deployment workflow (for example .github/workflows/backend-appservice.yml), verify the build and deploy paths match your backend folder exactly.

  • working-directory points to sprint4-cloud-deployment/backend.
  • Artifact upload path and deploy package path use sprint4-cloud-deployment/backend.
  • Deployment action uses secret APP_SERVICE_PUBLISH_PROFILE.
  • Node version aligns with App Service runtime (typically Node 20).
  • If no tests exist, remove or disable failing npm test steps.
- name: npm install, build, and test (backend only)
  working-directory: sprint4-cloud-deployment/backend
  run: |
    npm install
    npm run build --if-present

- name: Deploy to Azure Web App
  uses: azure/webapps-deploy@v3
  with:
    app-name: "<your-appservice-name>"
    package: sprint4-cloud-deployment/backend
    publish-profile: ${{ secrets.APP_SERVICE_PUBLISH_PROFILE }}

Frontend SWA Workflow Checks

For workflow such as .github/workflows/azure-static-web-apps.yml, check that the build points at the correct frontend directory and receives the backend URL secret at build time.

  • app_location: "sprint4-cloud-deployment/frontend"
  • output_location: "dist"
  • api_location: "" because backend is separate App Service
  • Deploy job environment includes VITE_BACKEND_API_URL secret mapping when needed
  • close_pull_request_job includes the same app_location if YAML validation requires it
env:
  VITE_BACKEND_API_URL: ${{ secrets.VITE_BACKEND_API_URL }}

with:
  app_location: "sprint4-cloud-deployment/frontend"
  api_location: ""
  output_location: "dist"

Deployment Trigger and Monitoring

After pushing changes, use GitHub Actions as your first diagnostics layer. A green workflow usually means the problem is in runtime configuration, while a red workflow usually means path, script, or secret issues.

  1. Push a commit to deployment branch (main by default).
  2. Open GitHub Actions and watch backend and frontend workflows.
  3. Confirm build, artifact upload, and deploy steps are green.
  4. If failed, open logs and use the troubleshooting page in this site.
Security reminder: Never commit publish profiles, API tokens, or any .env file to the repository.
