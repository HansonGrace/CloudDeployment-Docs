Azure Basics

This page consolidates how the course project uses Azure App Service, Azure Static Web Apps, and Azure MySQL so support staff can verify each service configuration quickly.

WVU policy reminder: For course projects, request Azure resource groups through Statler ITS instead of creating them directly.

Azure Service Roles in This Project

Each Azure service has a specific responsibility. Keeping these responsibilities separate makes it easier to locate issues quickly when a deployment partially works.

Azure Service Purpose Typical Ownership
Resource Group Container for all class-team cloud resources Provisioned through Statler ITS
Static Web App (SWA) Hosts built React frontend from GitHub Actions Instructor-provisioned, team-configured
App Service (Linux) Runs Node.js backend API Instructor-provisioned, team-configured
Azure Database for MySQL Flexible Server Stores users and backend data Provisioned earlier, reused in deployment

MySQL and SSL Requirements

The backend depends on a secure TLS connection to Azure MySQL. If SSL setup is incomplete, authentication and data routes can fail even when the app itself starts.

  1. In Azure Portal, open your MySQL Flexible Server networking page.
  2. Ensure firewall rules allow required client IP or cloud path per class setup.
  3. Download the SSL certificate from Azure portal.
  4. Place certificate at backend path config/DigiCertGlobalRootG2.crt.pem.
  5. Verify backend reads this CA cert for secure TLS connection.
const caPath = path.join(process.cwd(), "config", "DigiCertGlobalRootG2.crt.pem");

const pool = mysql.createPool({
  host: process.env.DB_HOST,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: process.env.DB_DATABASE,
  port: Number(process.env.DB_PORT || 3306),
  ssl: { ca: fs.readFileSync(caPath), rejectUnauthorized: true }
});

App Service Application Settings

In App Service go to Configuration > Application settings and verify:

Setting Expected value
DB_HOST <server>.mysql.database.azure.com
DB_USER Project database user
DB_PASS or DB_PASSWORD Project database password
DB_NAME or DB_DATABASE Project database name
FRONTEND_URL https://<your-swa>.azurestaticapps.net
JWT_SECRET Long random secret value
NODE_ENV production

After changes, save and restart the App Service.

App Service Runtime and Health

App Service controls runtime networking. Your Node app must respect Azure runtime values instead of local defaults to boot correctly in production.

  • Backend must use Azure assigned port: const PORT = Number(process.env.PORT) || 5175;
  • Do not hard-code production port values.
  • Validate health endpoint after deploy: https://<backend>.azurewebsites.net/health
  • Use App Service Log Stream for startup and database error analysis.

Static Web App Deployment Basics

The SWA build process compiles frontend code at deployment time. That means environment values must be available in GitHub Actions before deployment starts.

  • Azure-generated workflow should target sprint4-cloud-deployment/frontend.
  • Build output must be dist for Vite projects.
  • Provide backend URL through secret VITE_BACKEND_API_URL.
  • Confirm deployed frontend sends requests to https://<appservice>.azurewebsites.net/auth/....

Operational Checklist for Staff

  • Resource group exists and is mapped to correct course team.
  • App Service can start and return successful health check.
  • SWA deployment is green in GitHub Actions.
  • CORS is configured so SWA origin can call backend with credentials.
  • Backend can connect to MySQL without timeout or SSL errors.
